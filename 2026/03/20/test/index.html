    Java下的基本漏洞

    很多漏洞类别（注入、认证缺陷、XSS、CSRF、文件上传等等）在PHP与Java都存在，但实现细节、典型触发点、利用工具与防护手段往往不同。PHP常见的问题更倾向于“动态/弱类型与内置函数造成的危险”（eval/include/unserialize/文件包含、直接把输入当命令），而Java的风险更多来自“复杂生态与序列化/远程命名服务（JNDI）/第三方库漏洞（gadget chains）”以及框架配置错误（Spring、Struts、Jackson等）

    SQLi

    原理和php是一样的，发生情况就是没有使用PreparedStatement直接用了Statement.execute()执行；ORM/框架错误的使用原生SQL

    ORM框架

    ORM Object-Relational Mapping 对象关系映射

    在java中我们是面向对象编程，采用面向对象的方式来操作数据库就很统一，而且这在一定程度上防止了sql注入

    之前在学校学的是MyBatis(一种半自动化ORM)，上例子

    MyBatis 中#{ }是预编译参数(PreparedStatement占位符 MyBatis 会通过JDBC的PapredStatement.setXXX()方法把参数安全地传入数据库)，可以有效防止SQL注入；而${ }是字符串替换，直接把值拼到SQL中，只有在必须拼表明、列明、排序方向等无法用参数化时才使用，要做白名单校验。

    XSS

    不同于PHP大部分都是原生拼接，Java有很多不同的书写语言 JSP/Servlets、Thymeleaf、Freemarker、Spring MVC等。另有JSP EL、Spring表达式（SpEL）不当使用会更危险。

    CSRF

    与PHP原理相同——浏览器自动带cookie导致跨站请求，Java的Spring Security 默认提供CSRF保护，但一些REST API 可能故意禁用

    防护：双重提交/CSRF token 对JSON APIs 使用same-site cookie等。

    反射

    概念：是Java中的一种机制，它允许程序在运行时动态地访问类的结构和行为，包括类、方法、字段、构造函数等，从而实现动态调用和操作

    作用：运行时动态获取类型信息（通过反射，可以获取某个类的完整信息，例如其类名、父类接口、接口、构造函数、方法和字段等）；动态操作对象的属性和方法（即使在编译器并不知道具体的类和方法名，反射也可以在运行时访问和调用这些元素）；绕过访问限制（通过反射，可以访问私有字段和方法XXX.setAccessible(true)）

    modefiers

    modefiers是JDK自带的java.lang.reflect.Field类的一个私有成员变量，它在早期的Java版本（JDK8及之前）用于存储该字段（Field对象）自身的修饰符信息。也就是说，Field类自己内部有一个类似这样的定义

    在java反射机制中，即使通过setAccessible(true)绕过了访问权限检查，final修饰符依然会阻止修改字段值，所以我们要先用反射拿到这个字段的Field对象->再用反射去修改这个Field对象内部的modifiers值，去去掉它的final标志位。

    ClassLoader

    Java虚拟机（JVM）并不能直接识别我们写的.java源文件，而是需要通过javac命令将它们转换成.class文件（字节码文件）后才能执行。

    每个类加载器都有一个父加载器，且父加载器不是父类

    一个ClassLoader创建时如果没有指定parent，那么它的parent默认就是AppClassLoader

    Java中的三个系统类加载器

    Bootstrap ClassLoader(引导类加载器)：Java中最顶层的类加载器（启动类加载器，C++语言实现，是虚拟机自身的一部分），负责加载核心类库。它会加载%JRE_HOME%\lib目录下的重要文件

    Extension ClassLoader(扩展类加载器)：拓展类加载器用于加载%JRE_HOME%\lib\ext目录下的.jar包和.class文件。除此之外，还可以通过 JVM 参数 -Djava.ext.dirs=路径 指定其他扩展加载目录。

    Application ClassLoader(应用类加载器)：也被称为 System ClassLoader，它是用户最常接触的类加载器，主要负责加载当前应用程序的类路径（classpath：.;%JAVA_HOME%\lib;%JAVA_HOME%\lib\tools.jar）中的所有类。用户自己编写的类和依赖库大多由它加载

    三个类加载顺序为从上到下，分别是查阅sun.boot.class.path,java.ext.dirs和java.class.path来加载资源文件的。

    ClassLoader的类继承关系图

    三个系统类加载器之间的关系

    JVM初始化sun.misc.Launcher并创建Extension ClassLoader和AppClassLoader实例。并将Extension ClassLoader设置为AppClassLoader的父加载器。Bootstrap没有父加载器，但是他却可以作用一个ClassLoader的父加载器，比如Extension ClassLoader，而因为Boostrap不是java语言书写是虚拟机的一部分，在Extension调用getParent()时返回的内容为null。

    双亲委托

    一个类加载器查找class和resource时，是通过“委托模式”进行的，它首先判断这个class是不是已经加载成功，如果没有的话它并不是自己进行查找，而是先通过父加载器，然后递归下去，直到Bootstrap ClassLoader，如果Bootstrap classloader找到了，直接返回，如果没有找到，则一级一级返回，最后到达自身去查找这些对象。这种机制就叫做双亲委托。

    双亲委托机制分为向上委托和向下委托两部分：

    一个AppClassLoader查找资源时，先看看缓存是否有，缓存有从缓存中取出，否则委托给父加载器，一直这样递归向上直到Bootsrap;

    如果Bootstrap缓存没有找到的话，就去自己的规定路径下（sun.mic.boot.class）找，找到就返回，没有就向下让子加载器找（ExtClassLoader的java.ext.dirs->AppClassLoader的java.class.path，还没有就会抛出各种异常）

    实现双亲委托的重要方法

    loadClass

    JDK文档中是这样写的，通过指定的全限定类名加载class，它通过同名的loadClass(String,boolean)方法。

    上面是方法原型，一般实现这个方法的步骤是

    1. 执行findLoadedClass(String)去检测这个class是不是已经加载过了。
    2. 如果没有，就向上调用父加载器的loadClass;如果父加载器为null，则调用Bootstrap ClassLoader。
    3. 如果向上委托父加载器没有加载成功，则通过findClass(String)查找。

    defineClass

    当我们需要从D盘某个文件加载一个class文件，因为这个文件不是我们指定目录下的jar包或资源，所以如若我们要加载它，就需要自定义一个classLoader

    自定义步骤：

    1. 编写一个类继承自ClassLoader抽象类。
    2. 复写它的findClass()方法。
    3. 在findClass()方法中调用defineClass()(这个方法能将class二进制内容转换成Class对象)。

    要按照需求改写findClass()并调用defineClass去生成Class

    代理

    静态代理与动态代理的对比：

    特性 静态代理 动态代理
    代理类 手写 JVM 运行时生成
    接口要求 可以没有接口 必须有接口（JDK 动态代理的前提）
    扩展性 改一点逻辑就得重写类 一个 InvocationHandler 搞定
    典型用途 结构简单 AOP、RPC、ORM、Spring 代理机制等

    静态代理

    在编译时期就已经确定了代理类的实现。代理类需要实现与目标对象相同的接口，并持有目标对象的引用，通过代理对象调用目标对象的方法。

    动态代理

    必须要定义接口，不需要写实现类和代理类，由JVM自动动态生成一个“代理类”。它的实现主要依赖java.lang.reflect.Proxy和InvocationHandler

    ASM字节码

    看懂字节码

    Class file version：确定改写/运行环境的版本

    Constant pool：包含字节码中几乎所有的引用（字符串、类、方法、字段描述）

    字段(fields)与属性(attributes)：字段可能有ConstantValue属性（把常量值放到常量池，并让编译器优化使用它）

    方法字节码(Code)：ASM修改的直接对象

    • aload_0：把局部变量 0（通常是 this）压栈（引用类型）

    • aload_1 / aload：加载引用局部变量

    • astore：把栈顶引用存入局部变量

    • iload / istore：整型加载/存储

    • ldc：把常量池中的常量（字符串、int、float、Class 等）推到栈上

    • getfield：从对象取出实例字段，返回到栈

    • putfield：将栈顶值写入对象字段

    • getstatic / putstatic：静态字段访问

    • new：创建对象（仅分配内存）

    • dup：复制栈顶（构造器用）

    • invokespecial：用于构造器、私有方法、父类方法调用

    • invokevirtual：普通实例方法调用

    • invokestatic：静态方法调用

    • invokeinterface：接口方法调用

    • invokeDynamic：动态调用（bootstrap）

    • areturn：返回引用类型值

    • ireturn：返回整型

    • return：无返回值

    LineNumberTable/LocalVariableTable：调式与定位注入点；LocalVariableTable列出方法的局部变量名、槽位、作用范围（start/length）。对ASM插桩定位局部变量非常有用

    descriptors(描述符)与signature(签名)

    • 方法描述符(Ljava/lang/String;)V表示参数是String，返回void
    • Signature属性用于泛型源码签名(比如List)，在字节码中保留更精确的类型信息（供反射使用）

    class为Java8格式（ASM API 选Opcodes.ASM5/6/7/8/9都可以兼容）

    ACC_PUBLIC, ACC_SUPER为类访问标志

    interfaces: 0, fields: 1, methods: 2, attributes: 1 类有1个字段，两个方法（构造器与getMessage()），attributes包含SourceFile

    #2 = String #21 表示常量池里有个字符串常量 “Initial Value”（#21 是 Utf8）。

    #3 = Fieldref 指向类 Example 的字段 message 的符号引用。
    常量池是你用 ASM 插桩或手工生成指令时引用的目标（会用到 visitLdcInsn, visitFieldInsn，并提供描述符如 Ljava/lang/String;）

    #8 = Utf8 ConstantValue意味着编译器将其视作编译时常量，所以getMessage()用ldc返回常量

    构造函数

    构造器将常量池的字符串推入并存入字段putfiled。可以在ASM中修改这里的ldchuoputfield来改变默认值。

    getMessage()

    getMessage()并没有aload_0+getfield message，而是直接ldc将常量返回。说明编辑器因为字段是常量(ConstantValue)而将方法体编译为直接返回常量值(编译器/优化行为)

    LineNumberTable：把字节码指令映射到源代码行，方便调试与 ASM 插桩定位。

    LocalVariableTable：显示局部变量名字（this 等）。

    JNDI基本概念

    JNDI（Java Naming and Directory Interface）Java命名和目录接口。J2EE规范的重要组成部分，它为Java应用程序提供了一种命名和目录服务的统一接口，允许程序员查找和使用各种资源（如数据库、远程对象等）

    Naming：通过键值对的方式为对象创建唯一标识，使得应用程序可以通过名称检索这些对象。

    Directory：不仅允许通过名称检索对象，还支持基于属性的搜索。

    不使用JNDI的开发方式

    这种方式配置耦合，灵活性不足，不利于扩展

    使用JNDI的开发方式

    通过JNDI，可以将这些配置从程序中抽离出来，由容器进行管理，开发人员只需通过名称引用资源

    JNDI技术细节及代码示例

    1. StateFactory与ObjectFactory
      • StateFactory负责保存对象的状态（类似于持久化操作）。
      • ObjectFactory负责从状态信息中恢复对象实例。允许将储存在命名或目录服务中的对象转换为Java中的对象。
    2. SPI机制
      • 通过jndi.properties文件配置服务提供者接口（SPI）。
      • 支持灵活扩展，按需加载适配不同的协议和实现。
    3. 容器资源管理
      • 从J2EE 1.3开始，资源管理职责从应用程序转移到容器。
      • 应用程序通过<resource-ref>引用资源，容器负责解析和管理具体配置。
    • 使用RMI registry作为InitialContext的provider
    • 使用LDAP集中命名与提供目录服务的
    • 还有容器管理的JNDI（Tomcat\WildFly等等）

    Reference

    用于JNDI存放“引用”（类名与工厂信息）而非具体对象，当客户端lookup时，JNDI会使用ObjectFactory或类记载来重构或加载实际对象。这个机制是JNDI可远程加载类的根源（也正是很多安全问题的入口）

    目的：有些类非常的大或者需要通过某种特殊方式才能创建出来，如果每次都加载，负担重，于是采用工厂方法，每次使用的时候引用工厂，让工厂去生成它。

    三要素：对象类名、工厂类名、地址（可选）

    实现MyDataSourceFactory的getObjectInstance()，返回真正的DataSource.

    JNDI安全管理器

    Java中的对象分为本地对象和远程对象，本地对象是默认为可信任的，远程对象是不受信任的。比如当我们的系统从远程服务器加载一个对象，为了安全起见，JVM会限制该对象的能力（like禁止该对象访问我们本地的文件系统等），这些就是依靠安全管理器来实现的。

    JVM中采用的最新模型见上图，引入了“域”的概念，在不同的域中执行不同的权限。JVM会把所有代码加载到不同的系统域和应用域，系统域专门负责与关键资源进行交互，而应用域则通过系统域的部分代理来对各种需要的资源进行访问，存在于不同域的class文件就具有了当前域的全部权限。

    安全管理器架构

    JNDI注入

    通常发生在应用程序允许用户输入并通过JNDI查找对象时。攻击者可能通过构造恶意JNDI查找请求，诱使应用加载不安全的远程对象，进而执行恶意代码。

    要想成功利用JNDI注入，重要的前提就是当前JDK的版本，而JDK注入中不同的攻击向量和利用方式所被限制的版本号有点不一样。

    这里将所有不同版本JDK的防御都列出来：

    • JDK 6u45、7u21之后：java.rmi.server.useCodebaseOnly的默认值被设置为true。当该值为true时，将禁用自动加载远程类文件，仅从CLASSPATH和当前JVM的java.rmi.server.codebase指定路径加载类文件。使用这个属性来防止客户端VM从其他Codebase地址上动态加载类，增加了RMI ClassLoader的安全性。
    • JDK 6u141、7u131、8u121之后：增加了com.sun.jndi.rmi.object.trustURLCodebase选项，默认为false，禁止RMI和CORBA(公共对象请求代理体系结构)协议使用远程codebase的选项，因此RMI和CORBA在以上的JDK版本上已经无法触发该漏洞，但依然可以通过指定URI为LDAP协议来进行JNDI注入攻击。
    • JDK 6u211、7u201、8u191之后：增加了com.sun.jndi.ldap.object.trustURLCodebase选项，默认为false，禁止LDAP协议使用远程codebase的选项，把LDAP协议的攻击途径也给禁了。

    因此，我们在进行JNDI注入之前，必须知道当前环境JDK版本这一前提条件，只有JDK版本在可利用的范围内才满足我们进行JNDI注入的前提条件。

    RMI表攻击客户端

    将恶意的Reference类绑定在RMI注册表中，其中恶意引用指向远程恶意的class文件，当用户在JNDI客户端的lookup()函数参数外部可控或Reference类构造方法的classfactoryLocation参数外部可控时，会使用户的JNDI客户端访问RMI注册表中绑定的恶意Reference类，从而加载远程服务器上的恶意class文件在客户端本地执行，最终实现JNDI注入攻击导致远程代码执行。

    因为一般我们是构造恶意的uri并诱导客户端去访问它，从而加载恶意类并执行里面的内容，所以就有了这和SSRF好像的想法。实际上这种利用一定意义上是一种SSRF，如果有SSRF漏洞，JNDI可以成为利用它的一种。

    那还有一个疑问就是，直接访问恶意类不行吗？为啥要用Reference？这个当然是不行了，直接访问确实可能会让浏览器把class当作二进制文件下载，但不会在JVM上自动执行，Reference就是一个告诉访问的人怎么加载并实例化这个恶意类的向导。目标通过JNDI/LDAP/RMI取得Reference，然后自动去codebase下载并执行类字节码。

    基于Weblogic T3\IIOP协议的远程命令执行漏洞(CVE-2023-21839)

    影响版本：

    Oracle Weblogic Server 12.2.1.3.0

    Oracle Weblogic Server 12.2.1.4.0

    Oracle Weblogic Server 14.1.1.0.0

    漏洞成因

    Weblogic t3/iiop协议支持用JNDI将远程对象bind到服务端，进行远程绑定时实现了OpaqueReference接口，在进行lookup需要实体化时，会调用这个对象的getReferent函数进行查询。而碰巧有一个名为ForeignOpaqueReference的对象，它的getReferent函数在进行远程对象查询的时候，会再次发起JNDI查询，从而造成了JNDI注入。

    利用步骤

    1. 建立一个恶意ForeignOpaqueReference对象，并将remoteJNDIName设置为远程恶意JNDI服务。
    2. 通过T3\IIOP协议在WLS上绑定该恶意对象。
    3. 通过lookup查询该恶意对象，触发ForeignOpaqueReference.getReferent的调用，从而造成恶意JNDI注入。

    RMI原理及常见反序列化攻击手法

    RMI三层架构

    客户端：

    存根/桩（Stub）：远程对象在客户端上的代理

    远程引用层（Remote Reference Layer）：解析并执行远程引用协议

    传输层（Transport）：发送调用、传递远程方法参数、接受远程方法执行结果。

    服务端：

    骨架（Skeleton）：读取客户端产地的参数方法，调用服务器的实际对象方法，并接收方法执行后的返回值

    远程引用层：处理远程引用后向骨架发送远程方法调用

    传输层：监听客户端的入站链接，接受并转发调用到远程引用层

    注册表：以URL形式注册远程对象，并向客户端回复远程对象的引用。

    代码解析

    我们在使用RMI远程方法调用的时候，需要事先定义一个远程接口，继承java.rmi.Remote接口。该接口仅为RMI表示接口，本身不代表任何使用方法，说明可以进行RMI java虚拟机调用。

    远程接口实现类必须继承UnicastRemoteObject类，用于生成Stub和Skeleton

    Stub可以看作远程对象在本地的一个代理，囊括了远程对象的具体信息，客户端可以通过这个代理和服务端进行交互。

    Skeleton可以看作服务端的一个代理，用来处理Stub发送过来的请求，然后去调用客户端需要的请求方法，最终方法执行结果返回给Stub

    RMIServer将提供的服务注册在了RMIService上，并且公开了一个固定的路径，供客户端访问。
